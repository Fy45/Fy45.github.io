Work with interledger sdk(ETH)

Hi all, using interledger, we are supposed to link blockchains, traditional banking system and the moblie money system together.

Here’s how this streaming technology actually works. The problem still up here is we are trying to transfer value from one ledger to the other. There’s a litany(一连串) of different solutions to solve this interoperability prolem. In the general case, we want to be able to do is devise an algorithm (创造一个算法) that could enable the transfer of value from one ledger to another without them knowing anything about each other, without us knowing anything about the facilitator of this transaction or the risk preferences of any of the parties involved.

And to do this, we leverage(利用) the interledger protocol (ILP) with a setup that looks something like this in the figure.

sender, receiver, and the connector who’s basically the facilitator of this transaction here, we’re doing it from ETH and BTS, but it could be any base ledger.

What we do is we start off with an algorithm that sets the parameters for value transfer between these two different currencies. base unit are drastically different XRP(10^-6^) ETH(10^-18^). So from the respective ETH, the base unit value from XRP is massive.

You have to negotiate this problem first, done automatically.

Eth的最小单位是Wei, 一个Ether需要10^18^个wei来组成

XRP的最小单位是drop, 一个XRP需要10^6^个drops来组成

BTS的精度是5，一个BTS需要10^5^个base unit

Then exchange rate that calculated, as long as the sender agrees with the same exchange rate, we are off to the payments.

3 steps process of how the streaming payment is actually performed

  1. Negotiation of packet size/ exchange rate/ slippage b/w these two unknown ledgers, with the ability to intelligently monitor this as time goes on.

  2. Synchronous process where the streaming actually begins. We can see it in the console log (between connector and receiver), what stream does it that it sends this max packet size from the connector to the receiver. and this acts as a mechanism to draw up the balance held by the connector which means as time goes on, the connector is owing the receiver more and more money.

    收款方不会让connector欠他们无限的（infinite）的款项，但是connector自己是不知道这个ultimate value 是多少

    stream通过最终hitting a wall with its max packets size,并且得到一个max packet size error

    但是stream并不停止， 它会渐渐的reduce the packet size down直到到一个合适的amount of credit extended by the reciver.

  3. Parallel process of drawing the account down via payment channel updates, and these 2 independent processes work in tandem(串联) until the containment is complete.

    1. streaming process increases the account balance from connector to receiver,
    2. payment channel updates act to draw down the balance. (requires a cryptographic signature)

==Meaningful point==

Integrate more ledgers into the system with the ultimate goal of enabling a user to make it just in time payment from one currency to the other.

Imagine how a user could hold one specific cryptocurreny and access a service built on top of ethereum such as the smart contract, and be able to access that down to an arbitrarily(随意的) small value.
